CLAUDE.md best practices
1
Use /init to bootstrap Start here
Never write CLAUDE.md from scratch. Run /init inside Claude Code — it scans your project, infers the stack, and generates a starter file. Then edit what it got wrong rather than writing from zero.
/init
→
Claude scans codebase
→
CLAUDE.md generated
→
You refine it
Do
Run /init in every new CAP project
Review and trim what it generates
Re-run /init after major refactors
Don't
Copy-paste from another project
Keep everything /init generated blindly
Write it by hand from scratch
2
Purge aggressively Most important
After /init, delete anything that Claude would know anyway (how JavaScript works, what OData is, basic CDS syntax). CLAUDE.md is for your project's specific decisions — not a tutorial for Claude.
Keep
Your specific directory layout
Non-obvious naming decisions
Rules that differ from defaults
Commands unique to your project
Purge
"CAP uses CDS for data models"
"Use async/await not callbacks"
Technology introductions & overviews
Anything in Claude's training data
Rule of thumb: If you'd find it in the official SAP CAP docs, delete it from CLAUDE.md. Claude already knows it. Only write what's unique to your team or project.
3
Keep it under 200 lines Performance
Every line of CLAUDE.md consumes context window tokens on every single request. A bloated file means less room for your actual code. Treat each line as having a cost.
Idealunder 100 lines
Acceptable100–200 lines
Warning zone200–400 lines
Problematic400+ lines
If you're over 200 lines: move specific content to .claude/rules/ files that load on demand. Keep CLAUDE.md as a short index, not a full manual.
Good — concise rule
## Critical Rules - No raw SQL — CAP OData only - @restrict in .cds, not handlers - Handlers → srv/handlers/
Bad — over-explained
## Critical Rules We never write raw SQL because CAP's OData layer provides safe abstraction. This means you should always use SELECT.from() or cds.run() and never use hdb or hana-client directly because it bypasses...
4
Only universally applicable content Quality
CLAUDE.md is loaded on every request. Only put content that applies to every task Claude does in this project. Deployment steps, security audit checklists, and database migration guides don't belong here — they're only needed occasionally.
Always relevant → CLAUDE.md
Directory layout (db/, srv/, app/)
Naming conventions used everywhere
Error handling pattern (req.error)
Core dev command (cds watch)
Indent, quotes, semicolons
Occasional → move to rules/
Full deployment checklist
Security audit procedures
HANA migration steps
GDPR annotation full guide
Multi-tenancy setup steps
The test: Before adding a line to CLAUDE.md, ask — "will Claude need this when writing a simple GET handler?" If no, it doesn't belong in CLAUDE.md.
5
Move specific content to .claude/rules/ On demand
Topic-specific rules live in separate files under .claude/rules/. Claude loads them only when the task matches — so your security rules don't bloat every coding request.
.claude/rules/
├── security.md ← loaded when auth / data access work
├── deployment.md ← loaded when deploy / build tasks
├── database.md ← loaded when schema / migration work
├── multitenancy.md ← loaded when MT-related tasks
└── gdpr.md ← loaded when PII / data privacy work
# .claude/rules/security.md ## Auth rules (loaded for security tasks) - @restrict must be present on every exposed entity - Never trust client-provided user identity fields - req.user.id is the only trusted identity source - Service keys never hardcoded — use BTP bindings - Audit logging required for PII entity access
6
Split for modularization Scalability
As projects grow, one monolithic CLAUDE.md becomes unmanageable. Split by concern into separate files. CLAUDE.md becomes a short index that references them — Claude only loads what it needs.
Before — monolithic
# CLAUDE.md (380 lines) ## Architecture 40 lines ## Code Style 30 lines ## Libraries 20 lines ## Commands 25 lines ## Security Rules 60 lines ## Deploy Steps 80 lines ## DB Migration 70 lines ## GDPR Guide 55 lines ← all loaded on every request
After — modular
# CLAUDE.md (85 lines) ## Architecture 40 lines ## Code Style 30 lines ## Commands 15 lines @import .claude/rules/security.md @import .claude/rules/deployment.md @import .claude/rules/database.md @import .claude/rules/gdpr.md ← imports loaded on demand only
Split trigger: When any section grows past ~25 lines, it's a candidate to extract. A section that's only relevant to some tasks is always a candidate.
7
Use @import to reference external docs Composition
@import lets CLAUDE.md reference other markdown files without inlining their content. Claude loads the imported file only when relevant — keeping the main file lean while making specialised knowledge available.
# CLAUDE.md — my-btp-cap-app ## Architecture db/ → CDS models | srv/ → handlers | app/ → Fiori ## Code Style JS · 2 spaces · single quotes · no semicolons Naming: PascalCase entities, camelCase handlers ## Commands dev: cds watch | build: cds build --production deploy: cf deploy gen/my-app.mtar # ↓ loaded on demand when task matches @import .claude/rules/security.md @import .claude/rules/deployment.md @import .claude/rules/database.md @import .claude/rules/gdpr.md @import docs/api-standards.md ← external doc @import docs/architecture-decision-records.md
Good @import targets
Security & auth rules
Deployment checklists
Architecture Decision Records
API standards docs
BTP service config guides
Don't @import
Very large docs (100+ pages)
Things Claude already knows
Files that change constantly
Duplicate content from CLAUDE.md
8
Subdirectory CLAUDE.md files Precision
You can place a CLAUDE.md inside any subdirectory. When Claude works in that folder, it reads the subdirectory file in addition to the root one. This lets each layer of your BTP CAP project give Claude targeted instructions.
my-btp-app/
├── CLAUDE.md ← root: architecture + global style
├── db/
│ └── CLAUDE.md ← db rules: entity naming, aspects, no logic
├── srv/
│ └── CLAUDE.md ← srv rules: handler pattern, req.error, cds.log
├── app/
│ └── CLAUDE.md ← app rules: Fiori annotations, UI5 patterns
└── test/
└── CLAUDE.md ← test rules: cds.test(), mocking, coverage
db/CLAUDE.md
## DB layer rules - Use cuid + managed aspects always - PascalCase entity names - camelCase property names - No business logic in db/ - Associations not compositions unless true ownership
srv/CLAUDE.md
## Service layer rules - One handler file per entity - module.exports = (srv) => {} - req.error() not throw - cds.log() not console.log - Async/await only - No direct DB in handlers
How Claude reads them: Root CLAUDE.md sets project context. When Claude edits a file in srv/, it also reads srv/CLAUDE.md and merges both. The subdirectory file overrides the root for conflicts.
9
Commit CLAUDE.md to git Never lose it
CLAUDE.md is as important as your source code. If you don't commit it, the next developer (or the next machine) starts from scratch — and Claude gives different answers because it has no context.
# .gitignore — what to commit vs ignore # ✅ ALWAYS commit these CLAUDE.md .claude/rules/ .claude/commands/ .claude/agents/ .claude/skills/ # ❌ NEVER commit these .claude/settings.json ← may contain local paths .env default-env.json .claude/cache/
Good git practices
Commit CLAUDE.md alongside feature work
Update it when architecture changes
Review it in PRs like any other code
Add rules/ and commands/ to git
Avoid
Leaving CLAUDE.md in .gitignore
Stale CLAUDE.md that doesn't match code
Committing secrets inside CLAUDE.md
One person owns it — everyone edits it
Team tip: Treat CLAUDE.md changes in PRs seriously — they change how Claude behaves for the whole team. A wrong rule added here affects everyone's AI-assisted coding.
10
Keep it fresh — regular hygiene Maintenance
A CLAUDE.md that drifts from the actual codebase is worse than no CLAUDE.md — it gives Claude wrong context confidently.
Review triggers
After a major refactor
When a library is swapped out
When directory structure changes
When a new BTP service is added
Quarterly — even if nothing changed
Warning signs
Claude suggests deprecated patterns
Claude puts files in wrong directories
CLAUDE.md mentions removed libraries
Commands in CLAUDE.md no longer work
