.claude folder
The .claude/ folder is Claude Code's local config directory. It controls how Claude behaves — either for one project or across every project on your machine.
Directory structure
~/project-root/
│ CLAUDE.md← main instruction file, read at session start
│ .claude/← config directory
│ ├── settings.json← project-level tool & env config
│ ├── commands/← reusable slash commands
│ ├── agents/← sub-agent instruction files
│ └── skills/← modular SKILL.md task guides
~/.claude/← global config (all projects)
├── CLAUDE.md← your personal defaults everywhere
└── settings.json← global tool preferences
Scope — where configs live
Global
~/.claude/
Applies to every project on your machine. Great for personal code style, preferred language, and tools you always want enabled.
+
Project
./project/.claude/
Applies only inside that project. BTP CAP architecture rules, SAP libraries, and project-specific commands live here.

CLAUDE.md is the first thing Claude Code reads when it opens your project. Think of it as the briefing document you hand Claude before it touches any code.
How Claude reads it
Session starts
→
Reads ~/.claude/CLAUDE.md
→
Reads ./CLAUDE.md
→
Merges into context
→
Starts coding
Six sections
Architecture
Directory layout — db/, srv/, app/, handlers/. Claude places new files correctly without guessing.
Code style
Language, indent, quotes, naming conventions (PascalCase entities, camelCase handlers), error patterns.
Preferred libraries
SAP packages to use (@sap/cds, @sap/xssec) and libraries to explicitly avoid.
Commands
cds watch, mbt build, cf deploy — so Claude generates runnable commands, not guesses.
Critical rules
Hard constraints: no raw SQL, @restrict in CDS, no secrets in repo. Never violated.
Technical constraints
Node version, @sap/cds pin, HANA Cloud vs SQLite, XSUAA vs IAS auth provider.
Pro tip: Keep CLAUDE.md under ~200 lines. If it grows longer, split concerns into Skills — Claude loads those on demand rather than on every session.
Custom commands are reusable slash commands stored as markdown files in .claude/commands/. Type /command-name in Claude Code to run them.
How it works
/review
→
.claude/commands/review.md
→
Claude executes instructions
File structure
.claude/commands/
├── review.md← triggered by /review
├── deploy.md← triggered by /deploy
├── test-service.md← triggered by /test-service
└── scaffold-entity.md← triggered by /scaffold-entity
Example — /review command
# .claude/commands/review.md ## Code Review Checklist $ARGUMENTS ← receives file path or text after /review Review the file above for: - [ ] No raw SQL — all DB access via CAP OData layer - [ ] @restrict annotations present on all entities - [ ] Handlers in srv/handlers/, not inline in .cds - [ ] req.error() used, not raw throw - [ ] No hardcoded credentials or secrets - [ ] cds.log() used instead of console.log
Common BTP CAP commands
/scaffold-entity
Generates a CDS entity + service + handler scaffold following your project conventions. Accepts entity name as argument.
/deploy-cf
Runs cds build → mbt build → cf deploy in sequence with pre-flight checks. Prompts if workspace is dirty.
/add-odata
Adds a new OData action/function to the named service, wires up handler, and writes a Jest test stub.
/review
Runs the full BTP CAP code review checklist against the file passed as argument.
Tip: Commands support $ARGUMENTS — text typed after the slash command is injected at that placeholder. Use it for entity names, file paths, or feature descriptions.
Rules are hard constraints embedded in CLAUDE.md under a ## Critical Rules section. Claude treats them as non-negotiable — no exceptions, no workarounds.
Where rules live in CLAUDE.md
## Critical Rules - ⚠️ Never bypass the CAP OData layer — no raw SQL or direct DB calls - ⚠️ All access control via @restrict in .cds files, not in handlers - ⚠️ Service handlers go in srv/handlers/, never inline - ⚠️ Always use req.user for identity — never trust client-provided fields - ⚠️ Never commit .env, default-env.json, or service keys
Rule categories for BTP CAP
Data layer
No raw SQL. No direct HANA client calls. All reads and writes go through the CAP OData service layer.
Security
Authorization
@restrict annotations in .cds files only. Never re-implement auth logic in handlers. Use req.user, never trust client data.
Auth
Secrets
Never commit .env, default-env.json, or service keys. Credentials live in BTP service bindings only.
Compliance
Structure
Handlers in srv/handlers/. No logic inline in .cds files. Single-responsibility per service file.
Architecture
Error handling
Use req.error() for user-facing errors. cds.log() for diagnostics. Never raw throw or console.log in production.
Quality
Multitenancy
Tenant isolation via @sap/mtxs. Never share DB connections across tenants. Feature flags via BTP, not hardcoded env vars.
MT
Rules vs style: Rules go in ## Critical Rules — Claude treats them as absolute. Preferences (indent, quotes) go in ## Code Style — Claude follows them but may adapt in context.
Agents are sub-Claude instances that Claude Code can spin up to handle specialised tasks in parallel or in sequence. Each agent has its own instruction file in .claude/agents/.

→
Orchestrator Claude

→
Merges results
Agent directory
.claude/agents/
├── cds-modeler.md← specialises in CDS data modelling
├── fiori-builder.md← specialises in Fiori Elements config
├── test-writer.md← writes Jest tests for CAP services
└── security-reviewer.md← audits auth and data exposure
Example agent file
# .claude/agents/cds-modeler.md ## Role You are a CDS data modelling specialist for SAP CAP projects. ## Responsibilities - Design CDS entity definitions in db/schema.cds - Add @title, @description, and @UI annotations - Define associations and compositions correctly - Apply @PersonalData for GDPR-relevant fields ## Constraints - Never add business logic — that belongs in srv/handlers/ - Always use managed aspects (cuid, managed) unless told otherwise - Follow naming: PascalCase entities, camelCase properties
Common BTP CAP agents
cds-modeler
Designs entity schemas, associations, annotations, and aspects. Keeps db/ clean and focused on data shape only.
fiori-builder
Writes Fiori Elements annotations — @UI.LineItem, @UI.FieldGroup, value helps, and page configurations.
test-writer
Writes Jest tests for CAP service handlers. Knows cds.test() patterns, mocking, and BTP service stubs.
security-reviewer
Audits for missing @restrict, exposed fields, weak auth patterns, and GDPR annotation gaps.
When to use agents: Agents shine on large tasks that have clearly separable concerns — model the schema (cds-modeler) while simultaneously writing the Fiori annotations (fiori-builder) and test stubs (test-writer).
Skills are modular SKILL.md files loaded by Claude only when they are relevant to the current task — not on every session. This keeps the main CLAUDE.md lean.
Skills vs CLAUDE.md
CLAUDE.md
Always loaded. Contains project-wide architecture, style, and rules that apply to every task Claude does.
Always on
SKILL.md
Loaded on demand. Contains task-specific best practices — only consumed when Claude is about to do that type of task.
On demand

# .claude/skills/cap-handler/SKILL.md ## CAP Event Handler Conventions ### File location All handlers: srv/handlers/{EntityName}.handler.js ### Pattern module.exports = (srv) => { srv.before('CREATE', 'Orders', async (req) => { ... }) srv.on('READ', 'Orders', async (req) => { ... }) srv.after('CREATE', 'Orders', async (data, req) => { ... }) } ### Rules - Use req.error(400, 'msg') — never throw - req.user.id for identity - cds.log() not console.log
Custom skills: Write your own SKILL.md files for recurring task types — generating test data, writing OpenAPI specs, setting up BTP destinations. Claude reads and follows them exactly like built-in skills.
settings.json controls which tools Claude Code can use, environment variables, and model config. Lives in .claude/settings.json (project) or ~/.claude/settings.json (global).
Settings structure
// .claude/settings.json { "model": "claude-sonnet-4-20250514", "tools": { "allowed": ["bash", "read_file", "write_file", "search"], "blocked": ["browser"] ← disable tools you don't want }, "env": { "CDS_ENV": "development", "NODE_ENV": "development" }, "permissions": { "allowNetworkAccess": false, "allowFileSystemAccess": true } }
Key settings for BTP CAP
Tool allowlist
Restrict Claude to only the tools your project needs. Block browser access in secure CI pipelines. Allow bash for cds commands.
Environment vars
Inject CDS_ENV, NODE_ENV, and other dev-time vars. These are safe non-secret config — never put service keys here.
Model selection
Pin to a specific model version for consistent output across your team. Useful for reproducible CI code generation.
Permissions
Control filesystem and network access. Tighten for production-adjacent environments, loosen for local dev workflows.
Global vs project: Set your personal defaults globally (~/.claude/settings.json) — preferred model, your coding style env vars. Override per-project for BTP-specific tool restrictions or CI pipeline settings.
