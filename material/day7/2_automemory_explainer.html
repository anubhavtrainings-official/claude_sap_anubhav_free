Auto-memory & /memory
what is auto-memory
Auto-memory defined Claude Code
Auto-memory is Claude Code's system for automatically capturing and persisting context across sessions. When Claude notices something worth remembering — a preference you expressed, a pattern you corrected, a decision you made — it silently writes that into a memory store so future sessions already know it without you repeating yourself.
the problem it solves
Without auto-memory — you repeat yourself every session
Session 1
Always use req.error() — never throw raw errors in our CAP project.
Got it, I'll use req.error() for all error handling.
Session 2 — next day
throw new Error('Not found') ← Claude forgot. You need to say it again.
Session 3 — a week later
throw new Error('Unauthorized') ← Still forgotten. Context is gone.
With auto-memory — Claude remembers across sessions
Session 1
Always use req.error() — never throw raw errors.
Noted. Saving to memory: "Use req.error() for all CAP errors."
Session 2 — next day
req.error(404, 'Record not found') ← Already knows. No reminder needed.
Session 3 — one month later
req.error(401, 'Unauthorized') ← Still remembers. Context persists.

two ways to write to memory
1. Automatic Passive
Claude silently writes to memory when it detects corrections, strong preferences, or important decisions — without you doing anything.
# You say naturally in conversation: "Don't use console.log — we use cds.log()" "We always add the managed aspect to entities" "Our handlers export module.exports = (srv) =>" # Claude captures these without being asked
2. Manual via /memory Explicit
Use the /memory command to explicitly write, view, or delete entries — giving you full control over what persists.
# Explicitly save something: /memory add "Always use cds.test() for CAP tests" /memory add "Project uses HANA Cloud, not SQLite" /memory add "Auth: XSUAA with @sap/xssec middleware" # Claude confirms and stores immediately

/memory command reference
/memory — full command reference
/memory
Opens the memory panel — shows all current entries, their source (auto/manual), and timestamps. Your full memory at a glance.
/memory add "..."
Explicitly adds a new memory entry. Put the text in quotes. Claude confirms the save immediately. Best for architectural decisions you want guaranteed to persist.
/memory delete [id]
Removes a specific entry by its ID. Use this to purge stale memories when your project changes — e.g. after switching from SQLite to HANA Cloud.
/memory clear
Wipes all memory entries for the current project. Use carefully — typically only when starting a major refactor or when memory has drifted from reality.
/memory export
Exports all memory entries as a markdown file. Useful for reviewing what Claude knows, onboarding teammates, or seeding a new project's CLAUDE.md.
practical /memory workflow for BTP CAP
# Start of a new CAP project — seed memory manually /memory add "Runtime: Cloud Foundry on BTP" /memory add "DB: SAP HANA Cloud — not SQLite in any env" /memory add "Auth: XSUAA with JWT — never mock auth in staging" /memory add "Handlers: module.exports = (srv) => {} pattern" /memory add "Never use console.log — always cds.log()" # Review what's stored /memory # After migrating from SQLite to HANA /memory delete sqlite-dev-entry-id /memory add "Dev DB now also HANA Cloud via cds bind" # Export to review or use in CLAUDE.md /memory export
Pro tip: Run /memory export periodically and paste the important entries into your CLAUDE.md. That way the context is available to teammates too — not just your local Claude instance.
auto-memory vs CLAUDE.md
Auto-memory
Written automatically as you work
Captures corrections and preferences mid-session
Personal — lives on your machine only
Evolves automatically as the project changes
Not shared with teammates
Not version controlled in git
Can drift if not reviewed
vs
CLAUDE.md
Written intentionally and reviewed by humans
Committed to git — shared with the whole team
Structured by section — easy to audit in PRs
Source of truth for project architecture rules
Requires manual effort to write and maintain
Doesn't capture session-by-session learnings
Static until someone deliberately updates it
Best practice — use both together
They're complementary, not competing. The ideal workflow is:
Auto-memory captures learnings as you code day to day — zero friction, no manual work.
Periodically run /memory export and review what got captured.
Promote important entries to CLAUDE.md so teammates benefit too — not just you.
CLAUDE.md stays the team-level source of truth, memory handles your personal session continuity.
benefits of auto-memory
Zero repetition
Say a preference once. Auto-memory ensures Claude carries it into every future session — no re-briefing required at the start of each conversation.
Gets smarter over time
Every correction you make improves Claude's future behaviour. The longer you use it in a project, the better it understands how your team works.
Instant project context
New sessions start with full context loaded — Claude knows your stack, your patterns, your rules — without reading the whole codebase again.
Fills CLAUDE.md gaps
Things you forgot to put in CLAUDE.md get captured automatically as you encounter them — memory is the safety net for the things you didn't think to document.
Personal layer on top
CLAUDE.md is the team layer. Memory is your personal layer — individual preferences that are yours, not necessarily the team's convention.
Decision audit trail
Memory entries carry timestamps. You can see when you made a technical decision — useful for understanding why something was done months later.
especially powerful for SAP BTP CAP
BTP service binding patterns — once Claude learns your specific xs-security.json structure and mta.yaml layout, it applies them every session without guidance.
Annotation style decisions — whether you use @UI.LineItem inline or in a separate annotations.cds file, Claude remembers your team's preference.
HANA-specific quirks — things like which HANA features your version supports, or specific HANA Cloud instance naming conventions, persist across sessions.
Fiori element preferences — table vs list report, chart configurations, filter bar layouts — auto-memory means Claude matches your UI patterns without needing them in CLAUDE.md.
Error correction persistence — every time you correct a CAP-specific mistake (wrong handler syntax, wrong OData version assumption), that correction is permanent going forward.
The compound effect: After 2–3 weeks of active use, auto-memory plus CLAUDE.md gives Claude a level of project awareness that would take hundreds of lines of documentation to replicate manually — and it stays current automatically.
