  C1. resetPassword action does nothing

  Where: srv/handlers/user-management.handler.js:46-55
  Issue: The handler validates input and returns true without ever hashing or
  persisting the new password.
  Why it matters: Admins (and integration clients) call POST 
  /user-management/resetPassword expecting the user's credentials to be updated.
   The endpoint claims success but the password is unchanged. This is a silent
  security failure ‚ÄĒ a locked-out user appears reset, but their actual stored
  hash is still the old one.
  Fix:
  // before
  srv.on('resetPassword', async (req) => {
      const { userId, newPassword } = req.data;
      if (!newPassword) return req.error(400, 'Password must not be empty');
      if (newPassword.length < 8) return req.error(400, 'Password must be at 
  least 8 characters');

      const user = await SELECT.one.from(Users).where({ ID: userId });
      if (!user) return req.error(404, 'User not found');

      return true;
  });

  // after
  const password = require('../auth/password');
  const UserCredentials = 'anubhav.claude.UserCredentials';

  srv.on('resetPassword', async (req) => {
      const { userId, newPassword } = req.data;
      if (!newPassword || newPassword.length < 8) {
          return req.error(400, 'Password must be at least 8 characters');
      }

      const user = await SELECT.one.from(Users).where({ ID: userId });
      if (!user) return req.error(404, 'User not found');

      const passwordHash = await password.hash(newPassword);
      const creds = await SELECT.one.from(UserCredentials).where({ user_ID:
  userId });

      if (creds) {
          await UPDATE(UserCredentials).set({ passwordHash, failedAttempts: 0
  }).where({ ID: creds.ID });
      } else {
          await INSERT.into(UserCredentials).entries({ user_ID: userId,
  passwordHash, failedAttempts: 0 });
      }
      return true;
  });

  W1. AuthService is wide-open with requires: 'any', and me() lives inside it

  Where: srv/auth-service.cds:1 ‚ÄĒ service AuthService @(path: 'auth', requires: 
  'any')
  Issue: login and refresh must be reachable anonymously; that's correct. But
  me() ‚ÄĒ which is defined for authenticated callers ‚ÄĒ sits in the same block and
   inherits requires: 'any'. The JS handler defends with if (!id || id === 
  'anonymous') req.error(401, ‚Ä¶), so security is preserved by accident.
  Why it matters: CDS-level guards should match handler-level expectations. A
  future refactor that drops the handler check will silently expose /auth/me.
  Fix:
  service AuthService @(path: 'auth') {
      @requires: 'any'
      action login(loginName: String, password: String) returns LoginResponse;

      @requires: 'any'
      action refresh(refreshToken: String) returns LoginResponse;

      @requires: 'authenticated-user'
      function me() returns UserInfo;
  }

  W2. Travellers.userID is a plain String(100) ‚ÄĒ no FK to Users

  Where: db/schema.cds:55-56
  Issue: userID is the link between a Traveller and the authenticating User.
  It's stored as free text. The cat-service.cds where: clauses (userID = 
  $user.email) rely on this matching Users.loginName exactly.
  Why it matters: A user rename, email change, or typo silently breaks @restrict
   filtering. There is no referential integrity.
  Fix:

  userID : String(100);

  // after ‚ÄĒ associate to Users; restrict by user.loginName
  user : Association to Users;
  // then in cat-service.cds:
  //   where: 'user.loginName = $user.id'   // or $user.email if you store it on
   Users

  W3. UserCredentials ‚Üí Users orphan risk (Association where Composition 
  belongs)

  Where: db/auth.cds:10-15
  Issue: UserCredentials.user : Association to Users is a plain reference, and
  Users has no composition pointing back. Deleting a User leaves the credentials
   row orphaned forever.
  Why it matters: Lifecycle of credentials is by definition bound to the user.
  This is the textbook case for composition (Mistake B in the data-model rules).
  Fix:
  // in db/schema.cds, add to Users
  entity Users : cuid {
      // ‚Ä¶existing fields‚Ä¶
      credentials : Composition of one UserCredentials on credentials.user =
  $self;
  }

  // in db/auth.cds ‚ÄĒ keep the back-association
  entity UserCredentials : cuid {
      user : Association to Users;
      // ‚Ä¶
  }

  Where: srv/handlers/auth.handler.js:31-35

  Why it matters: The schema implies brute-force protection that isn't actually
  wired. Either implement (e.g. lock at 5 failures by setting Users.isLocked =

  Fix:
  const MAX_FAILED = 5;

  Where: srv/handlers/user-management.handler.js:58-66
  Issue: No check that the user exists, no check that (user_ID, role_code)

   role_code: roleId });

  W10. JWT secret regenerates on every dev restart

  Where: srv/auth/jwt.js:5-11
  Issue: When AUTH_JWT_SECRET is unset (dev), a random secret is generated per

  Why it matters: It's a deliberate tradeoff (no shared dev key), but for a
  training app where students restart often it'll cause confusing "why am I
  logged out?" experiences. Persist a dev secret in .env.local (gitignored)

  ūüĒĶ Suggestions

  S1. Use i18n keys for handler error messages

  Where: all handler req.error(‚Ä¶) calls (e.g. auth.handler.js:20, 24, 27, 34, 
  51, 57, 61, 62).
  The project ships four locales but error messages are English-only. Switch to
  message codes (req.error(401, 'INVALID_CREDENTIALS')) and put text in
  _i18n/messages.properties.

  S2. Add @assert.format for emails

  Where: db/schema.cds:43 (Travellers.email), db/schema.cds:87
  (Users.loginName).
  @assert.format: '^[^@\s]+@[^@\s]+\.[^@\s]+$'
  email : String(120);

  S3. Add @PersonalData aspects for GDPR audit

  Where: Travellers.email, Travellers.phone, Travellers.firstName/lastName,
  Users.loginName/firstName/lastName.
  CAP ships cds.PersonalData annotations specifically for this. Worth doing
  once; future you (or auditors) will thank you.

  S4. Collapse trivial service entry files

  Where: srv/auth-service.js, srv/cat-service.js, srv/user-management.js are
  5-line files that just require and call one handler. Either inline the handler
   or accept the indirection ‚ÄĒ but be consistent. If you keep the pattern, drop
  the srv/handlers/*.handler.js suffix (.handler.js is redundant inside a
  handlers/ folder).

  S5. auth-service.cds is missing a logout action

  A token-revocation list is non-trivial with stateless JWT, but at minimum
  offer a logout action that clears any client-side state and (eventually) lands
   a refresh-token blocklist. Stub the endpoint now so consumers don't bake "no
  logout" into clients.
